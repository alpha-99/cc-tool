# 001 - 核心功能需求规格
## 版本: 1.0.0
## 最后更新: 2026-03-20

---

## 用户故事

### 故事 1: 初始化项目配置
**作为** 一个开发者
**我想要** 快速为我的项目初始化Claude Code配置文件
**以便** 我能立即开始使用Claude Code进行AI辅助开发

### 故事 2: 语言特定的模板
**作为** 一个使用特定编程语言的开发者
**我想要** 根据我的项目语言获取相应的配置文件模板
**以便** 配置符合该语言的最佳实践和工具链

### 故事 3: 安全的文件操作
**作为** 一个谨慎的开发者
**我想要** 工具在复制文件时避免覆盖现有配置
**以便** 不会意外丢失我已有的设置

### 故事 4: 清晰的反馈
**作为** 一个工具使用者
**我想要** 看到详细的操作日志和明确的错误信息
**以便** 了解工具的执行情况和诊断问题

### 故事 5: 用户级配置框架
**作为** 一个首次使用Claude Code的用户
**我想要** 自动创建用户级的Claude Code配置目录结构
**以便** 我可以在全局级别自定义Agent、命令和技能

## 功能性需求

### FR-001: 命令行接口
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-001.1 | 工具必须接受两个必需参数：`PROJECT_DIR`（项目目录路径）和 `LANGUAGE`（编程语言类型） | 必须 |
| FR-001.2 | 支持以下命令行选项：<br>- `--help`, `-h`: 显示帮助信息<br>- `--version`, `-v`: 显示版本信息<br>- `--list-languages`, `-l`: 列出支持的语言类型<br>- `--list-templates`, `-t`: 列出可用模板<br>- `--dry-run`, `-n`: 预览操作但不实际执行<br>- `--verbose`, `-V`: 输出详细调试信息<br>- `--quiet`, `-q`: 静默模式，只输出错误 | 必须 |
| FR-001.3 | `LANGUAGE` 参数不区分大小写，支持语言别名映射（js→javascript, py→python, golang→go） | 必须 |

### FR-002: 模板系统
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-002.1 | MVP阶段仅支持内置模板，存储在 `templates/<language>/` 目录 | 必须 |
| FR-002.2 | 每个语言模板目录必须包含以下结构：<br>- `.claude/` 目录及其子目录：<br>  - `settings.json` (必须)<br>  - `agents/` (可选，空目录)<br>  - `commands/` (可选，空目录)<br>  - `hooks/` (可选，空目录)<br>  - `skills/` (可选，空目录)<br>- `CLAUDE.md` (必须)<br>- `constitution.md` (可选)<br>- `specs/` 目录 (可选，空目录) | 必须 |
| FR-002.3 | 初始版本必须支持以下语言：python, javascript, go, rust, java | 必须 |
| FR-002.4 | 工具必须根据 `LANGUAGE` 参数查找对应的内置模板目录 | 必须 |

### FR-003: 文件复制行为
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-003.1 | 将模板目录中的所有文件和子目录递归复制到 `PROJECT_DIR` 中 | 必须 |
| FR-003.2 | 保持原始目录结构（如 `.claude/settings.json` → `<PROJECT_DIR>/.claude/settings.json`） | 必须 |
| FR-003.3 | **跳过已存在的文件**：如果目标文件已存在，不执行任何操作（不覆盖、不修改） | 必须 |
| FR-003.4 | 自动创建必要的父目录（如 `.claude/` 目录） | 必须 |
| FR-003.5 | 符号链接应作为普通文件复制，不保持符号链接属性 | 必须 |

### FR-004: .gitignore 规则管理
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-004.1 | 工具必须检查目标项目是否已有 `.gitignore` 文件 | 必须 |
| FR-004.2 | 如果 `.gitignore` 不存在，则创建并添加以下规则：<br>```<br># Claude Code 本地配置<br>.claude/settings.local.json<br><br># IDE 配置目录<br>.vscode/<br>.idea/<br><br># 操作系统文件<br>*.DS_Store<br><br># 项目生成目录（根据语言不同）<br>__pycache__/<br>*.pyc<br>node_modules/<br>target/<br>dist/<br>build/<br>``` | 必须 |
| FR-004.3 | 如果 `.gitignore` 已存在，则确保包含上述规则（可追加到文件末尾） | 必须 |

### FR-005: 模板变量替换
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-005.1 | 支持在模板文件中使用变量占位符，格式为 `{{VARIABLE_NAME}}` | 必须 |
| FR-005.2 | 支持的变量：<br>- `{{PROJECT_NAME}}`: 项目名称（从目标目录名自动提取）<br>- `{{PROJECT_DIR}}`: 项目目录的绝对路径<br>- `{{LANGUAGE}}`: 编程语言类型 | 必须 |
| FR-005.3 | 所有文本文件（`.json`, `.md`, `.txt` 等）中的变量都应被替换 | 必须 |
| FR-005.4 | 二进制文件（图片、压缩包等）不进行变量替换 | 必须 |

### FR-006: 输出与日志
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-006.1 | 默认输出详细的操作日志，包含：<br>- 正在初始化的项目目录<br>- 使用的语言模板<br>- 复制的每个文件<br>- 跳过的每个文件<br>- 完成统计 | 必须 |
| FR-006.2 | 日志格式示例：<br>`[INFO] 正在初始化项目: ./myproject`<br>`[INFO] 使用语言模板: python`<br>`[INFO] 复制: .claude/settings.json → ./myproject/.claude/settings.json`<br>`[INFO] 跳过已存在文件: ./myproject/CLAUDE.md`<br>`[INFO] 完成！共复制 3 个文件。` | 必须 |

### FR-007: 错误处理
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-007.1 | 必须显式处理以下错误情况：<br>1. `PROJECT_DIR` 不存在或不是目录<br>2. 权限不足（无法读取模板或写入目标）<br>3. `LANGUAGE` 参数不被支持<br>4. 找不到对应语言的模板目录<br>5. 文件系统I/O错误 | 必须 |
| FR-007.2 | 错误信息必须清晰明确，帮助用户诊断问题 | 必须 |
| FR-007.3 | 所有错误都应通过异常捕获处理，避免程序崩溃时输出堆栈跟踪（除非使用 `--debug` 标志） | 必须 |

### FR-008: 用户级框架初始化
| ID | 需求描述 | 优先级 |
|----|----------|--------|
| FR-008.1 | 工具在首次运行时，应检查并初始化用户级的Claude Code配置框架（如果不存在） | 必须 |
| FR-008.2 | 检查的目录和文件：<br>- `~/.claude/` 目录<br>- `~/.claude/commands/` 目录<br>- `~/.claude/skills/` 目录<br>- `~/.claude/agents/` 目录<br>- `~/.claude/settings.json` 文件<br>- `~/.claude/CLAUDE.md` 文件 | 必须 |
| FR-008.3 | 初始化行为：<br>- 如果目录不存在，创建空目录<br>- 如果文件不存在，创建包含基本注释的模板文件<br>- 如果目录或文件已存在，跳过（不覆盖、不修改） | 必须 |

## 非功能性需求

### NFR-001: 架构解耦
| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| NFR-001.1 | **模块化设计**：代码必须模块化，便于添加新的语言支持 | - 添加新语言模板时，无需修改核心复制逻辑<br>- 语言特定的配置与通用逻辑分离 |
| NFR-001.2 | **清晰的依赖关系**：所有依赖必须通过函数参数或对象成员显式注入 | - 不使用全局变量传递状态<br>- 函数签名明确显示所有依赖 |

### NFR-002: 错误处理质量
| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| NFR-002.1 | **显式错误处理**：错误绝不应该被默默忽略 | - 所有可能的异常情况都有明确的处理逻辑<br>- 错误信息包含足够的上下文供用户诊断 |
| NFR-002.2 | **防御性编程**：验证输入参数的合法性和边界条件 | - 在操作前验证目录存在性和可访问性<br>- 验证模板文件的完整性和可读性 |

### NFR-003: 性能要求
| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| NFR-003.1 | **响应速度**：能快速处理包含少量文件的项目 | - 初始化一个包含10个文件的模板应在2秒内完成 |
| NFR-003.2 | **资源效率**：递归复制时避免深度嵌套导致的性能问题 | - 处理嵌套深度不超过5层的目录结构不应有明显性能下降 |

### NFR-004: 可维护性
| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| NFR-004.1 | **代码清晰度**：代码的首要目的是让人类易于理解 | - 使用符合Python风格的代码（Pythonic）<br>- 避免不必要的复杂抽象 |
| NFR-004.2 | **可扩展性**：便于未来添加新功能 | - 模板查找逻辑支持未来添加用户自定义模板<br>- 变量替换系统支持未来添加新变量类型 |

### NFR-005: 可测试性
| ID | 需求描述 | 验收标准 |
|----|----------|----------|
| NFR-005.1 | **单元测试覆盖**：所有核心功能都必须有单元测试 | - 命令行参数解析<br>- 模板查找逻辑<br>- 文件复制逻辑<br>- 错误处理逻辑 |
| NFR-005.2 | **测试策略**：优先使用表格驱动测试（Table-Driven Tests） | - 核心逻辑测试使用表格驱动测试用例<br>- 测试覆盖率不低于80% |

## 依赖与约束

### 技术约束
1. **Python版本**: 必须支持 Python 3.11+
2. **跨平台**: 支持 Linux、macOS 和 Windows（通过WSL或原生）
3. **依赖最小化**: 优先使用Python标准库，外部依赖需明确理由

### 设计约束
1. **遵循项目宪法**: 所有实现必须遵循项目宪法中的三条核心原则
2. **测试先行**: 所有功能必须先编写失败的测试（TDD）
3. **简单性原则**: 只实现本规范中明确要求的功能（YAGNI）

### 兼容性约束
1. **向后兼容**: MVP版本不承诺向后兼容性
2. **向前兼容**: 模板系统设计应支持未来扩展（用户自定义模板、更多变量类型等）
